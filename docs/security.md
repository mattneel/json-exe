# Security model

JSON.exe runs JavaScript that may be authored by users or generated by an LLM.
Treat the threat model seriously.

## The critical rule

> `new Function()` is not a sandbox.

The default evaluator is named `"unsafe-new-function"` on purpose. Code it
compiles runs with the full authority of the host process. JSON.exe does **not**
claim that arbitrary untrusted code is safe merely because it only receives
`ctx`.

```ts
createRuntime({ evaluator: "unsafe-new-function" }); // the default — dev/trusted only
```

Use the default evaluator for:

- local development,
- trusted / admin-authored extensions,
- fast prototyping and tests.

For untrusted code (including prompt-injected, agent-generated extensions), run
an isolated evaluator: a Web Worker, a Node worker thread, a subprocess, or a
QuickJS/WASM VM. The `Evaluator` interface is the extension point:

```ts
interface Evaluator {
  readonly name: string;
  compile(input: { slot: string; source: string; async: boolean }): CompiledFn | Promise<CompiledFn>;
}
```

A production evaluator should provide: no ambient filesystem/network/process
access, wall-clock timeouts, CPU/memory budgets where available, frozen
intrinsics, a structured-clone boundary, and an immutable or proxied `ctx`.

## Capability-based context

Slots receive no authority except what the host puts in `ctx`. Context **is**
the capability boundary.

```ts
// Bad — hands the slot the keys to the kingdom:
ctx = { db, fs, fetch, process };

// Good — narrow, scoped, budgeted capabilities:
ctx = {
  input,
  user: safeUserView,
  tools: { search: limitedSearchTool, lookupCustomer: scopedCustomerLookup },
  budget: { toolCallsRemaining: 3 },
};
```

## What the runtime gives you (even with the dev evaluator)

- **Return validation** — every result is checked against its declared schema.
- **Structured errors** — stable `kind`s; error objects never embed `ctx`
  values, so secrets do not leak into traces or repair prompts.
- **`freezeContext`** — deep-freezes `ctx` (in place) before each run so slots
  cannot mutate the host's objects. Cycle-safe; skips accessor properties.
- **Best-effort timeouts** — `timeoutMs` (per-slot, default, or per-run). These
  are meaningful for async slots that yield; a synchronous infinite loop cannot
  be preempted under `new Function` and needs an isolated evaluator.
- **No automatic globals** — the runtime never injects Node globals into `ctx`.
  (A slot author can still reach ambient globals under `new Function`; that is
  the evaluator's responsibility to prevent, hence "unsafe".)

## Permissions

Extensions may declare a permission manifest:

```json
{
  "$permissions": { "tools": ["search"], "network": false, "memory": "read" }
}
```

Pass the host's actual grants as `grantedPermissions` when compiling; the
runtime checks declared ⊆ granted and raises `PermissionError` on over-reach
(SPEC §9.6). This is a declaration check — enforcing capability *use* at runtime
is the evaluator/`ctx`-design responsibility.

```ts
await compileExtension(spec, json, {
  grantedPermissions: { tools: ["search"] },
});
```

## Threats to keep in mind

Infinite loops, memory exhaustion, prototype pollution, data exfiltration,
access to globals/process/env/fs/network, malicious helper calls, expensive
computation, hidden side effects, and prompt-injected agent-generated
extensions. The dev evaluator mitigates almost none of these — isolation does.
